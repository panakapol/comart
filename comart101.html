Instruction Format — 32 bits ต่อ 1 คำ (Word)

โครงสร้าง OP-Code, Register, Address
💡

โจทย์จะให้ว่า "มี 32 bits ต่อ 1 คำ" แล้วถามว่า OP-Code มีกี่ bit / Register มีกี่ bit / เกี่ยวกับ bit ต่างๆ

1.1) OP-Code มีกี่ bit?

ดูจาก Instruction Format — ส่วน OP-Code คือบิตแรกของ instruction
ถ้า format = Opcode (4 bit) | Mode (1 bit) | Address (11 bit) รวม = 16 bit
สำหรับ 32-bit word ต้องดูจาก diagram ที่ให้มาในโจทย์

1.2) Register มีกี่ bit?

Register field = บิตที่ใช้ระบุตัว register (เช่น R0–R7 ต้องการ 3 bit)

1.3) เกี่ยวกับ bit ต่างๆ (วาดรูปเบอะ)

โจทย์อาจให้วาด Instruction Format diagram แสดง field ต่างๆ เช่น:

| OP-Code | I (Indirect) | Register | Address |

เลข bit ในแต่ละ field รวมกันต้องได้ 32 bits เสมอ

1.4) จำนวน / 1.5) อื่นๆ

อาจถามจำนวน instruction สูงสุดที่เป็นไปได้ = 2n (n = จำนวน bit ของ OP-Code)

OP-Code

ระบุประเภทของคำสั่ง เช่น ADD, LDA, STA
ถ้า n bit → มีได้ 2n คำสั่ง

Address Field

ระบุตำแหน่งหน่วยความจำที่จะเข้าถึง
เช่น 12 bit → address space = 212 = 4096

2

Register & AR (Address Register)

ตัวโหนดเกี่ยวกับ Register ในระบบ Common Bus

AR คือ Address Register — ใช้เก็บ ที่อยู่ (Address) ของหน่วยความจำที่จะอ่าน/เขียน

Registerชื่อเต็มหน้าที่
ARAddress Registerเก็บ address ของ memory ที่จะเข้าถึง
PCProgram Counterเก็บ address ของ instruction ถัดไป
IRInstruction Registerเก็บ instruction ที่กำลัง execute
DRData Registerเก็บข้อมูลที่อ่านมาจาก memory
ACAccumulatorเก็บผลลัพธ์ของการคำนวณ ALU
TRTemporary Registerเก็บข้อมูลชั่วคราว
🔑

AR ดูที่ติ๊กไว้ — AR มี input จาก: PC (ตอน Fetch), DR[0-11] (ตอน Indirect), และ IR[0-11] (ตอน Decode)
การที่ AR จะโหลดค่าได้ต้องมีสัญญาณ L (Load) และ clock อยู่ในสภาวะที่ถูกต้อง

โจทย์อาจถาม: AR โหลดค่าเมื่อไหร่?

ดูจาก Microoperations:
• R₀T₁: AR ← PC (ตอน Fetch)
• R₀T₂: AR ← 0, TR ← PC (ตอน Interrupt)
• D₅T₃ (Indirect): AR ← M[AR] (ถ้า I=1)
สัญญาณที่ทำให้ AR load = R₀T₁ OR D₅T₃ OR T₄ (...)

3

Microoperations — LDA / ADD

อธิบายว่าเกิดอะไรขึ้นในแต่ละ cycle

โจทย์จะให้ sequence ของ microoperation แล้วถามว่า "อธิบายว่าเกิดอะไรขึ้น"

Fetch Cycle (ทุก Instruction เริ่มที่นี่)
T₀: AR ← PC
T₁: IR ← M[AR], PC ← PC + 1
T₂: AR ← IR[0-11], I ← IR[15], Decode IR[12-14]
LDA (Load to Accumulator) — D₂
D₂T₄: DR ← M[AR]
D₂T₅: AC ← DR, SC ← 0

คือ: อ่านค่าจาก memory มาเก็บที่ DR แล้วโอนไป AC

ADD — D₁
D₁T₄: DR ← M[AR]
D₁T₅: AC ← AC + DR, E ← Cout, SC ← 0

คือ: อ่านค่าจาก memory → บวกกับ AC → ผลลัพธ์เก็บที่ AC, carry เก็บที่ E

⚠️

SC ← 0 หมายถึง Sequence Counter รีเซ็ต = จบ cycle นั้น กลับไปเริ่ม Fetch ใหม่

4

Prefix Expression → สมการทั่วไป

3-Address / แปลง Prefix เป็น Infix
วิธีแปลง Prefix → Infix
  • อ่าน Prefix จากซ้ายไปขวา
  • พบ Operand → push ลง stack
  • พบ Operator → pop 2 ค่าจาก stack, สร้าง (op1 operator op2), push กลับ
  • ทำซ้ำจนหมด → ผลใน stack คือคำตอบ
ตัวอย่าง: + A * B C → Infix
Prefix: + A * B C
อ่านจากขวา: C, B, * → (B * C)
ต่อ: A, + → (A + (B * C))
Infix: A + B * C
โจทย์จากเล่ม: ((A + B) + C) XOR ...

ต้องแปลงจาก Prefix กลับเป็น Infix พร้อมใส่วงเล็บให้ถูกต้องตาม precedence

Prefix (Polish)

Operator นำหน้า Operand
เช่น + A B = A + B

Postfix (Reverse Polish)

Operator ตามหลัง Operand
เช่น A B + = A + B

6

Pipeline — D[i] = A[i]*B[i] + A[i]*C[i]

Pipelined vs Non-Pipelined, Speedup

โจทย์ให้ operation: D[i] = A[i]*B[i] + A[i]*C[i] สำหรับ i = 1,2,3,...,7 (i=0 ด้วย)

6.1) วาดออกมาประมาณนี้ (Pipeline Diagram)

Pipeline 3 Segment:

Segment 1: R1 ← A[i], R2 ← B[i] (Load A, B)
Segment 2: R3 ← R1 * R2, R4 ← C[i] (Multiply & Load C)
Segment 3: R5 ← R3 + R4 (Add)
6.2) ความวิธีลดเวลา / 6.1) มาก

Pipeline ช่วยให้แต่ละ segment ทำงานพร้อมกัน (Concurrent) แทนที่จะทำทีละขั้น

6.3) Speedup Formula — สำคัญมาก ⭐
Non-Pipelined: τ = n × t_n
Pipelined (k stages): τ_p = (k + n - 1) × t_p

Speedup: S_k = (n × t_n) / ((k + n - 1) × t_p)

เมื่อ n → ∞: S_k → t_n / t_p = k (ถ้า t_n = k × t_p)

ตัวแปร: n = จำนวน tasks k = จำนวน stages t_p = clock cycle ของ pipeline

📌

จาก 6.2 และ 6.1 เทียบกัน — 6.2 ดีกว่า เพราะไม่ต้องเปรียบเทียบกับ 6.1 เหมือนกัน (Pipeline เร็วกว่า Non-Pipeline)

Assembly Language

เขียน Assembly จาก Pseudocode + แปลงเป็นฐาน 16
โจทย์ (Pseudocode ที่ให้มา)
i = -5, sum = 0
while (i ≤ 0) {
  if (i == -2 || i == -4)
    sum = sum + (-i - i) // บวกสองเท่า
  else
    sum = sum + (-i)
  i = i + 1
}
output = sum
7.1) เขียนเป็น Assembly (ตัวอย่าง Assembly ที่ให้มา)
; ตัวอย่าง Assembly โครงสร้าง
    SUM = 0
    C = -5
LOOP: while C <= 0
    if C ∈ {9,-2,54} ; ตามที่โจทย์กำหนด
      SUM = SUM - C
    end if
    C = C + 1
END: end while
7.2) แปลงจาก Assembly เป็นฐาน 16 (Hex)

แปลงค่าต่างๆ ใน Assembly เป็นเลขฐาน 16:

ค่า Decimalค่า Hex (ฐาน 16)ค่า Binary
00x000000 0000
-50xFB (2's complement)1111 1011
-20xFE1111 1110
-40xFC1111 1100

วิธีแปลง negative → Hex: หา 2's complement แล้วแปลงเป็น Hex

💡

โจทย์ถามเขียนโปรแกรม + Assembly + แปลงเลขฐาน — มีเขียน loop แล้วก็ if แปลงเลขฐาน
ข้อสอบมีประมาณ 7 ข้อ แล้วอาจจะพูดแต่ละครั้งไม่เหมือนเดิม อย่าถาม!

Pipeline Hazards (เนื้อหาเพิ่มเติม)

Major Hazards ใน Pipelined Execution
ประเภท Hazardอธิบายวิธีแก้
Structural Hazard
(Resource Conflicts)		 Hardware ไม่พอให้ทำพร้อมกัน
เช่น อ่าน memory 2 ที่พร้อมกัน		 เพิ่ม Hardware / Stall
Data Hazard
(Data Dependency)		 Instruction ต้องการผลจาก instruction ก่อน
ที่ยังคำนวณไม่เสร็จ		 Forwarding / Stall
Control Hazard
(Branch Address)		 Branch/Jump ทำให้ไม่รู้ว่า PC จะไปที่ไหน Branch Prediction / Flush
ตัวอย่าง Data Hazard
R1 ← R2 + R3 ; ยังไม่เสร็จ
R4 ← R1 + 1 ; ต้องการ R1 แต่ยังไม่ได้

แก้ด้วย Forwarding: ส่งผลจาก ALU ไปตรงๆ โดยไม่ต้องรอ writeback

🔴

Hazards ใน pipelines อาจทำให้ต้อง Stall pipeline (หยุดรอ) — ตรวจสอบด้วย Detect Hazard Logic
